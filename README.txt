마법소녀 슈팅: 던전 탐험 FPS
1. 프로젝트 개요
이 프로젝트는 90년대 고전 FPS의 감성을 현대적인 로그라이크 요소와 결합한 2.5D 레트로 FPS 게임입니다. 순수 JavaScript(Vanilla JS)와 HTML Canvas를 사용하여 개발되었으며, 별도의 게임 엔진 없이 레이캐스팅(Raycasting) 기법을 직접 구현하여 3D 환경을 렌더링합니다.

핵심 목표는 단순한 슈팅을 넘어, 무작위로 생성되는 던전을 '탐험'하고, 성장하며, 더 깊은 곳으로 도전하는 재미를 제공하는 것입니다.

2. 핵심 아키텍처: 계층별 상태 소유
이 프로젝트는 상태를 '무엇을 위한 것인가'에 따라 나누어, 각 계층이 자기 몫만 소유합니다.

Script/world.js: 시뮬레이션 상태의 단일 진실 공급원입니다. 플레이어, 적, 아이템, 맵 등
게임 세계를 이루는 데이터만 담으며, 전부 JSON으로 직렬화할 수 있습니다.
이 제약 덕분에 JSON.stringify(world) 한 줄로 세이브 파일이 만들어지고,
서브 던전에 들어갈 때 월드를 통째로 스택에 넣었다가 복원할 수 있습니다.
텍스처나 DOM 요소를 여기에 넣으면 이 성질이 즉시 깨집니다.

Script/runtime.js: 저장 대상이 아닌 세션 상태 (실행 중 여부, 화면 흔들림 위상,
무기 교체 애니메이션 진행 여부 등).

Script/assets.js: 로드된 텍스처, 사운드, 스프라이트 아틀라스, 던전 테마.
직렬화할 수 없고 게임 세션보다 수명이 깁니다.

Script/dom.js: HTML 요소 참조. 어떤 셀렉터로 무엇을 찾는지가 SELECTORS 표 한 곳에 모여 있어,
마크업을 바꿀 때 코드 전체를 뒤질 필요가 없습니다. 누락된 요소는 시작 시점에 한꺼번에 보고됩니다.

Script/actions.js: 세계 상태를 바꾸는 유일한 통로입니다.
체력 감소나 아이템 획득 같은 변경이 전부 이곳을 거치므로, 무적 판정이나 방어력 같은 규칙을
넣을 때 고칠 곳이 한 군데뿐입니다. 각 변경은 이벤트로 알려집니다.

Script/events.js: 게임 이벤트 버스입니다.
시뮬레이션은 "무슨 일이 일어났는가"만 알리고, 화면(ui.js)과 소리(audio.js)는 그것을 구독해
스스로 반응합니다. 덕분에 연출을 추가할 때 게임 규칙 코드를 열 필요가 없고,
gameLogic.js는 브라우저 없이도 실행·테스트할 수 있습니다.

이 구조가 주는 이점은 다음과 같습니다.

저장과 복원: world가 순수 데이터이므로 세이브/로드와 서브 던전 스택이 그대로 성립합니다.

추적 가능한 변경: 상태가 어디서 바뀌는지 actions.js만 보면 됩니다.

테스트 가능성: 시뮬레이션이 화면·소리·입력에 의존하지 않아 Node에서 헤드리스로 돌릴 수 있습니다.

느슨한 결합: 모듈 의존 그래프에 순환이 없습니다.

3. 주요 스크립트 파일 분석 (Script/)
파일명

역할

핵심 기능 및 설명

loop.js

고정 타임스텝 루프

시뮬레이션을 항상 SIMULATION_STEP_MS 단위로만 전진시킵니다. 화면 프레임에서 남은 시간은 누산기에 모아 다음 프레임으로 넘기므로, 30FPS든 144FPS든 같은 시간이 흐르면 같은 결과가 나옵니다.

main.js

게임 오케스트레이터 (진입점)

init()으로 모든 모듈을 초기화하고 각 계층을 이벤트에 연결한 뒤, requestAnimationFrame으로 메인 게임 루프를 실행합니다. 사망(PLAYER_DIED)과 출구 도달(EXIT_REACHED)을 구독해 게임 오버와 층 이동을 처리합니다.

world.js

시뮬레이션 상태

직렬화 가능한 게임 세계. createWorld/serializeWorld/deserializeWorld 제공.

runtime.js

세션 상태

저장하지 않는 실행 플래그와 연출용 값.

assets.js

에셋 저장소

텍스처, 사운드, 아틀라스, 던전 테마.

actions.js

상태 변경 통로

world를 바꾸는 유일한 경로이자 이벤트 발행 지점.

events.js

이벤트 버스

동기 Pub/Sub. EVENTS에 모든 이벤트 이름이 정의되어 있습니다.

dom.js

DOM 참조

SELECTORS 표 기반으로 요소를 찾고 누락을 한 번에 보고합니다.

constants.js

설정 및 밸런스 데이터

에셋 경로, 적 스탯, 무기 성능, 맵 크기 등 모든 정적 상수. 맵 타일의 성질(통행 가능 여부, 시야 차단, 벽 텍스처, 상호작용)도 TILE_TYPES 표 한 곳에 있어, 새 타일을 추가할 때 여러 파일을 고칠 필요가 없습니다.

render.js

렌더링 엔진 (그래픽)

레이캐스팅으로 벽/바닥/천장을, Z-버퍼로 스프라이트를 그립니다. 렌더 전용 버퍼는 이 모듈이 소유합니다.

gameLogic.js

게임의 두뇌 (로직)

이동과 충돌, 적 AI, 발사체, 공격 판정. 화면이나 소리를 알지 못합니다. 경로 탐색은 플로우 필드를 씁니다. 모든 적의 목표가 플레이어 하나이므로, 맵 전체의 "플레이어까지 걸음 수"를 한 번만 계산해 모두가 공유합니다. 적이 몇 마리든 경로 탐색 비용은 그대로입니다.

input.js

사용자 입력 처리

키보드/마우스/터치를 처리하고, 공격·상호작용은 '의도' 이벤트로 알립니다.

ui.js

HUD 및 UI 관리

게임 이벤트를 구독해 HUD와 연출을 갱신합니다.

audio.js

사운드 재생

게임 이벤트를 구독해 효과음을 재생합니다. 어떤 사건에 어떤 소리가 나는지는 표로 정의됩니다.

branches.js

던전 구조 (가지)

어떤 던전이 어디에서 갈라져 나오는지, 몇 층인지, 룬을 주는지를 정의한 표입니다. 메인 던전 아래로 짐승굴, 오크 광산, 보물창고, 심층부가 갈라지고 그 아래로 다시 늪지, 엘프 회관, 납골당, 지옥 등이 이어집니다. 새 던전을 추가하려면 이 표에 한 항목만 넣으면 입구 배치와 난이도 계산이 따라옵니다.

mapGenerator.js

레벨 디자이너

'방과 복도' 알고리즘으로 던전을 생성합니다. 전역 상태에 접근하지 않는 순수 함수입니다.

4. 데이터 및 에셋 관리 (Data/)
스프라이트 시스템: 모든 이미지는 타일시트(*.png)와 각 스프라이트의 좌표 정보를 담은
아틀라스 파일(*_data.json)의 쌍으로 이루어져 있습니다. 둘 다 Data/tiles/ 에 있습니다.

아틀라스는 손으로 쓰지 않고 생성합니다.

    npm run build:atlas

어느 시트의 어느 좌표를 무엇으로 쓸지는 tools/build-atlas.js 의 THEMES와 SPRITES 표에
모여 있습니다. 스프라이트를 교체하려면 이 표만 고치고 다시 생성하면 됩니다.
좌표가 시트 밖으로 나가면 생성 단계에서 오류로 멈추고, 코드가 요구하는 스프라이트가
빠지면 tests/atlas.test.js 가 실패합니다.

타일시트는 Dungeon Crawl Stone Soup 의 것을 사용합니다. 이 타일셋에는 Doom 식 HUD 얼굴
이미지가 없어 face_* 스프라이트는 비어 있으며, 해당 자리는 플레이스홀더로 표시됩니다.

화면 확인: 브라우저를 띄우지 않고도 한 프레임을 PNG로 뽑아 확인할 수 있습니다.

    npm run preview               (기본: main 테마)
    npm run preview -- cave 1     (테마와 변형 지정)

성능 측정: 레이캐스팅과 스프라이트 렌더링에 드는 실제 비용을 잽니다.

    npm run bench                 (기본 해상도)
    npm run bench -- 3 400        (스케일 3, 400프레임)

렌더 해상도는 constants.js 의 RENDER_RESOLUTION_SCALE 이 기본값이고,
실제 사용 값은 runtime.renderScale 에 있어 기기에 맞춰 조절할 수 있습니다.
값이 작을수록 선명하고 느립니다.

테마 시스템: Data/dungeon_progression.json 파일은 각 층의 테마(예: main, cave)와 변형(variation)을 정의합니다. render.js는 이 정보를 바탕으로 해당 층에 맞는 벽, 바닥, 천장 텍스처를 동적으로 불러와 사용하여 다채로운 던전 분위기를 연출합니다.

5. 게임 실행 흐름
초기화: main.js가 DOM 요소를 준비하고 모든 이벤트 리스너를 설정합니다.

에셋 로딩: 사용자가 'START' 버튼을 누르면 render.js의 loadAssets가 constants.js에 정의된 모든 이미지와 사운드를 비동기적으로 불러옵니다.

게임 시작:

main.js의 startGame 함수가 호출됩니다.

mapGenerator.js가 첫 번째 층의 맵 데이터를 생성합니다.

gameLogic.js가 생성된 맵에 적들을 스폰합니다.

게임 루프 (gameLoop) 시작:

입력: input.js가 사용자의 조작을 감지해 큐에 쌓습니다. 이 시점에는 게임 상태가 바뀌지 않습니다.

시뮬레이션: loop.js가 흐른 시간만큼 고정 크기 스텝을 실행합니다. 각 스텝의 시작에서 gameLogic.js가 입력 큐를 비우며 반영하고, 이어서 플레이어·적·발사체를 갱신합니다. 모든 상태 변경은 이 스텝 안에서만 일어납니다.

렌더링: render.js가 world의 정보를 읽어 3D 월드와 스프라이트를 캔버스에 그립니다.

UI 업데이트: ui.js가 HUD를 갱신하고, 피격·발사·무기 교체 같은 연출은 게임 이벤트를 구독해 처리합니다.

위 과정이 requestAnimationFrame을 통해 매 프레임 반복됩니다.

6. 서브 던전
메인 던전을 내려가다 보면 하위 던전으로 이어지는 입구를 만납니다. 어느 층에 어떤 입구가
놓일지는 게임 시작 시 한 번 정해지며(world.branchEntrances), 늪지와 해안처럼 짝을 이루는
가지는 한 판에 하나만 등장합니다.

서브 던전에 들어가면 지금 세계가 통째로 world.parentStack 에 보관되고 새 세계가 시작됩니다.
그 가지의 최하층에서 출구를 이용하면 보관해 둔 세계가 그대로 복원됩니다.
떠날 때의 지형과 적이 그 자리에 있고, 서브 던전에서 입은 피해와 얻은 룬은 따라옵니다.
스택은 중첩되지 않고 평평하게 쌓이므로 여러 단계를 내려가도 세이브 크기가 급격히 늘지 않습니다.

층 표시는 DCSS 관례를 따라 '가지:층' 형태입니다. D:3 은 메인 던전 3층, L:2 는 짐승굴 2층입니다.
적 스폰 수는 가지 안에서의 층이 아니라 누적 깊이를 기준으로 하므로,
짐승굴 1층은 메인 1층이 아니라 메인 10층 언저리의 위험도를 갖습니다.

7. 테스트
브라우저 없이 게임 로직을 검증할 수 있습니다.

    npm test

시드 난수와 가상 시계로 시뮬레이션을 1800프레임 재생하고, 그 결과를 tests/snapshot.json과
비교합니다. 이동, 벽 충돌, 적 추적, 발사체, 아이템, 무기 교체, 문 열기, 층 전환이 모두 포함되어
있어 동작이 조금이라도 달라지면 즉시 드러납니다.

게임 동작을 의도적으로 바꿨다면 아래로 기준 스냅샷을 갱신하고, git diff로 변화가
의도한 것인지 반드시 확인하십시오.

    npm run test:update-snapshot
